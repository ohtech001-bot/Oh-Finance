# ADR-0007 — استراتيجية المصادقة

**الحالة:** مُعتمد ومُنفَّذ · **التاريخ:** 2026-07-14 · **المرحلة:** 1

---

## السياق

نظام مالي. سرقة جلسة تعني قراءة بيانات زبائن، وتسجيل دفعات وهمية، وتعديل أرصدة.

المتطلب صريح: «لا تخزن Tokens الحساسة في localStorage.»

## القرار

### 1. الرموز في كوكيز HttpOnly — لا localStorage

| | localStorage | كوكي HttpOnly |
|---|---|---|
| XSS تسرقه؟ | **نعم** — أي JS يقرأه | **لا** — غير مرئي لـJS |
| يُرسل تلقائيًا؟ | لا | نعم ⇒ يحتاج حماية CSRF |

سطح هجوم XSS هائل: كودنا + **كل حزمة npm متعدّية**. ثغرة واحدة في أي منها تسحب الرمز من localStorage وترسله للمهاجم، الذي يستخدمه لاحقًا من جهازه.

كوكي HttpOnly لا يراه JavaScript. XSS تستطيع إطلاق طلبات نيابة عن المستخدم أثناء وجوده على الصفحة (سيّئ)، لكنها **لا تستطيع أخذ الجلسة والعودة بها**. الفارق جوهري.

**النتيجة:** `apps/web/src/lib/api.ts` لا يحتوي كلمة `localStorage` ولا أي متغيّر يحمل رمزًا.

### 2. حماية CSRF — Double-Submit

الكوكيز تُرسل تلقائيًا ⇒ موقع خبيث يستطيع إطلاق `POST /api/payments` بجلستك.

الحل: رمز CSRF في **موضعين** — كوكي **مقروء** + ترويسة `X-CSRF-Token`. الموقع الخبيث يستطيع إجبار المتصفح على إرسال الكوكي، لكنه **لا يستطيع قراءته** (Same-Origin Policy) فلا يملأ الترويسة.

`SameSite=Lax` طبقتنا الأولى، لكنها إعداد **متصفح** قد يسقط. الدفاع في العمق يعني ألا نعتمد على طبقة واحدة.

### 3. تدوير رمز التجديد + كشف إعادة الاستخدام

كل تجديد يُبطل الرمز القديم ويصدر جديدًا في نفس «العائلة».

```
1. مهاجم يسرق رمز تجديد.
2. المستخدم الشرعي يجدّد → الرمز القديم صار مُبطلًا.
3. المهاجم يستخدم الرمز المسروق (المُبطل).
4. ← استخدام رمز مُبطل. لا يحدث في الاستخدام الطبيعي أبدًا.
5. نُبطل العائلة كلها — بما فيها رمز المهاجم الجديد.
```

الثمن: خروج المستخدم الشرعي أحيانًا. المقابل: إغلاق الباب على السارق.

### 4. Argon2id لا bcrypt

bcrypt مقيّد بـ72 بايت (يقطع الباقي **بصمت**) وذاكرته ثابتة صغيرة ⇒ تكسره الـGPU بكفاءة.

Argon2id **صعب الذاكرة**: كل محاولة تخمين تحتاج 64 ميجابايت. GPU بآلاف الأنوية لا يشغّل إلا عشرات المحاولات المتوازية بدل الملايين. المعاملات تتبع OWASP (m=64MiB, t=3, p=4).

### 5. مقاومة تعداد المستخدمين — نصًا **وزمنًا**

توحيد الرسالة لا يكفي إن اختلف الزمن:

```
بريد غير مسجّل  → رد فوري    (~2ms)   ← لم نُشغّل Argon2
بريد مسجّل      → رد متأخر   (~100ms) ← Argon2 يعمل
```

المهاجم يقيس الزمن ويعدّ عملاءك بلا قراءة أي رسالة.

**الحل:** `verifyWithTimingEqualization()` — عند غياب المستخدم نتحقق من **هاش وهمي**، فيتساوى الزمن. ومسار «نسيت كلمة المرور» يعيد **نفس** الرسالة دائمًا.

### 6. طبقتا حماية من التخمين

- حد معدل: 5 محاولات/15د لكل IP → يوقف التخمين الموزّع على عدة حسابات.
- قفل حساب: 10 محاولات → قفل 15د → يوقف التخمين المركّز من عدة IPs.

كل طبقة وحدها قابلة للالتفاف؛ معًا لا.

### 7. الإبطال الفوري

`JwtAuthGuard` يفحص حياة الجلسة في قاعدة البيانات **مع كل طلب**. يكلّف استعلامًا، والمقابل: إيقاف محل، أو تعطيل موظف، أو كشف سرقة — يسري **فورًا**، لا بعد انتهاء الرمز.

## البدائل المرفوضة

| البديل | سبب الرفض |
|---|---|
| JWT في localStorage | XSS تسرقه. مرفوض بالمتطلب صراحةً. |
| JWT في الذاكرة + refresh في كوكي | يحمي من XSS جزئيًا، لكن كل تحديث للصفحة يتطلب دورة تجديد. تعقيد بلا مكسب حقيقي. |
| Session ID تقليدي (بلا JWT) | يتطلب استعلامًا لكل طلب — وهو ما نفعله أصلًا! الفارق أن الـJWT يحمل الصلاحيات فيوفّر استعلامًا ثانيًا. |
| رمز تجديد بلا تدوير | رمز مسروق يعمل حتى انتهائه. لا كشف. |
| bcrypt | حد 72 بايت + ضعف أمام GPU. |
