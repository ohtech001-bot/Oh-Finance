# ADR-0001 — عزل المستأجرين عبر PostgreSQL RLS

**الحالة:** مُعتمد · **التاريخ:** 2026-07-14 · **المرحلة:** 1

---

## السياق

نظام SaaS متعدد المحلات. بيانات كل محل (زبائن، طلبات، دفعات، حركات مالية) سرّية تمامًا عن بقية المحلات. تسرّب واحد = انتهاك ثقة لا يُصلَح.

النمط الشائع هو الفلترة في طبقة التطبيق:

```ts
prisma.customer.findMany({ where: { tenantId } })
```

هذا يعمل — حتى ينسى مطوّر واحد `where` مرة واحدة، أو يمرّ استعلام خام غير مفلتر، أو يُستغل ثغرة حقن. عندها يقرأ المستأجر كل شيء.

## القرار

**أربع طبقات دفاع، الحاسمة منها في قاعدة البيانات.**

| # | الطبقة | الآلية | التنفيذ |
|---|---|---|---|
| 1 | الهوية | `tenantId` من الـJWT الموقّع حصرًا | `token.service.ts` |
| 2 | السياق | `AsyncLocalStorage` طوال دورة الطلب | `tenant-context.ts` |
| 3 | الحارس | `JwtAuthGuard` هو المكان الوحيد الذي يُضبط فيه المستأجر | `jwt-auth.guard.ts` |
| 4 | **قاعدة البيانات** | **RLS بدور تطبيق بلا `BYPASSRLS`** | `0002_rls_and_hardening` |

### التفاصيل الحاسمة

**دور التطبيق `oh_app` بلا `BYPASSRLS`.** لو ملكه، لصارت كل السياسات زينة.

**`set_config('app.tenant_id', $1, true)` — على نطاق المعاملة.**
مع تجميع الاتصالات (Neon/PgBouncer)، إعداد على نطاق الجلسة (`SET`) يبقى عالقًا على الاتصال ويتسرّب إلى طلب مستأجر آخر يلتقط نفس الاتصال. المعامل `true` يجعله transaction-scoped فيُمحى عند `COMMIT`. الثمن: معاملة لكل طلب.

**`FORCE ROW LEVEL SECURITY`** — بدونه يتجاوز مالك الجدول السياسات بصمت.

**`WITH CHECK` على كل سياسة** — يمنع الكتابة بـ`tenant_id` مزوّر، ويمنع «تهريب» صف بتغيير مستأجره.

**ثقب واحد موثّق: `app_auth_lookup`.**
تسجيل الدخول يحدث قبل معرفة المستأجر (نعرف البريد فقط). دالة `SECURITY DEFINER` ضيقة تتجاوز RLS، تقبل بريدًا وتُرجع حقول المصادقة فقط — لا اسم، لا هاتف، لا بيانات أعمال.

**المدير العام لا يصل إلى بيانات الأعمال.**
`ROLE_PERMISSIONS.SUPER_ADMIN` صلاحيات منصة فقط. `PermissionsGuard` يرفض دخوله مسارات المحل. سياسة `platform_access` تتطلب `app.is_platform='on'` الذي لا يُضبط إلا في `runAsPlatform()` — ولا تستدعيها أي خدمة أعمال.

## البدائل المرفوضة

| البديل | سبب الرفض |
|---|---|
| فلترة بالتطبيق فقط | خطأ برمجي واحد = تسرّب كامل. لا حارس مستقل. |
| قاعدة بيانات لكل مستأجر | هجرات على آلاف القواعد؛ تكلفة اتصالات هائلة؛ تقارير عابرة للمستأجرين مستحيلة. |
| Schema لكل مستأجر | أخف، لكن `search_path` عرضة للحقن، والهجرات تظل مضاعفة. |
| RLS وحدها بلا حارس تطبيق | رسائل خطأ رديئة (0 صفوف بدل 403)، وتحميل غير ضروري على القاعدة. |

## العواقب

**إيجابية:** خطأ برمجي يُنتج «صفر صفوف» لا «كل الصفوف» — فشل آمن. مُختبَر بـ30 اختبار تكامل تقرأ **بلا `where` إطلاقًا**.

**سلبية:**
- معاملة لكل طلب HTTP ⇒ استهلاك اتصالات أعلى. (مخاطرة R7؛ مخفّفة بمهلات قصيرة.)
- كل جدول أعمال جديد **يجب** أن يحمل `tenant_id` + سياسة. النسيان يعني جدولًا مفتوحًا.
