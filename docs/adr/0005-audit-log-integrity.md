# ADR-0005 — سلامة سجل التدقيق (سلسلة هاش)

**الحالة:** مُعتمد ومُنفَّذ · **التاريخ:** 2026-07-14 · **المرحلة:** 1

---

## السياق

المتطلب: «سجل نشاط لا يمكن التلاعب به.»

منع `UPDATE`/`DELETE` عبر التطبيق سهل. لكن من يملك وصولًا مباشرًا لقاعدة البيانات — مسؤول نظام، أو مهاجم اخترق بيانات الاعتماد، أو من يستعيد نسخة احتياطية مسرّبة — يستطيع نظريًا تعديل صف وإعادة كتابته.

سجل يمكن تزويره ليس سجل تدقيق.

## القرار

**سلسلة هاش (hash chain) فوق جدول append-only.**

```
hash[n] = sha256( hash[n-1] ‖ tenant ‖ action ‖ entity ‖ actor ‖ payload ‖ timestamp )
```

كل قيد يحمل `prev_hash` (هاش سابقه) و`hash` (هاشه هو).

### لماذا يعمل هذا؟

تعديل أي صف يغيّر هاشه. لكن الصف التالي يخزّن الهاش **القديم** في `prev_hash` — فينكسر الربط. لإخفاء التعديل يجب إعادة حساب **كل** الهاشات اللاحقة: أثر ضخم، ويتطلب كتابة على جدول مسحوبة منه صلاحية الكتابة.

`audit:verify` يمسح السلسلة ويُبلّغ عن **أول موضع انكسار** — فنعرف أي قيد مُسّ ومتى.

### ثلاث طبقات

1. `REVOKE UPDATE, DELETE ON audit_logs FROM oh_app;`
2. Trigger `BEFORE UPDATE OR DELETE` يرفع استثناءً.
3. سلسلة الهاش — تكشف ما نجح في تجاوز الطبقتين.

### تفاصيل حاسمة

**قفل استشاري عند الكتابة.** بلا `pg_advisory_xact_lock`، يقرأ طلبان متزامنان نفس `prev_hash` فتتفرّع السلسلة وتصير غير قابلة للتحقق.

**تسلسل JSON بترتيب مفاتيح ثابت.** بدونه يختلف الهاش بين تشغيلين على نفس البيانات، فيفشل التحقق كذبًا.

**سلسلة مستقلة لكل مستأجر.** لا تتشابك سجلات المحلات.

**تنقيح إلزامي.** السجل يخزّن «قبل/بعد» لكل تعديل. بلا تنقيح، تعديلُ كلمة مرور يكتب هاشها في جدول **يقرأه صاحب المحل من شاشة سجل النشاط**.

**الكتابة داخل نفس معاملة العملية.** فشل العملية ⇒ تراجع قيد التدقيق معها. لا سجل لعملية لم تحدث، ولا عملية بلا سجل.

## البدائل المرفوضة

| البديل | سبب الرفض |
|---|---|
| جدول عادي + منع UPDATE/DELETE | يوقف التطبيق، لا من يملك وصولًا مباشرًا للقاعدة. |
| توقيع كل صف بمفتاح خاص | أقوى، لكن يتطلب إدارة مفاتيح (HSM/KMS). مؤجَّل — السلسلة كافية للكشف. |
| WORM خارجي (S3 Object Lock) | ممتاز للأرشفة، لكنه لا يحمي الجدول الحيّ الذي تقرأ منه الواجهة. |
| Blockchain | تعقيد هائل بلا فائدة إضافية هنا: لدينا سلطة مركزية موثوقة أصلًا. |

## العواقب

**إيجابية:** التلاعب **قابل للكشف** بشكل مثبت رياضيًا. مُختبَر بـ9 اختبارات، منها محاكاة تعديل صف قديم مباشرة.

**سلبية:** كل قيد تدقيق يقرأ سابقه (استعلام إضافي + قفل) ⇒ تسلسل الكتابات على السجل. مقبول: التدقيق ليس مسار العمليات الساخن.
