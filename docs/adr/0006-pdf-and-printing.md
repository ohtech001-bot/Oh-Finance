# ADR-0006 — استراتيجية PDF والطباعة

**الحالة:** مُعتمد (التصميم) · **التنفيذ:** المرحلة 6 · **التاريخ:** 2026-07-14

---

## السياق

المتطلبات 15–17: طباعة طلب، وعرض سعر/مسودة **بالعربية والعبرية** (`הצעת מחיר` / `טיוטה`)، وكشف حساب كامل.

وشاشة الإعدادات تعرض طابعة حرارية `Xprinter XP-58` بقياس `80mm`.

المشكلة التقنية: **تشكيل النص ثنائي الاتجاه**. العربية تتطلب:
- ربط الحروف حسب موضعها (أول/وسط/آخر/منفرد): `ع` تصير `عـ` / `ـعـ` / `ـع`
- الأربطة (ligatures): `لا` حرف واحد لا حرفان
- خوارزمية Unicode BiDi: أرقام لاتينية داخل نص عربي داخل جدول

خطأ واحد في أي منها ⇒ فاتورة بحروف مقطّعة أو مقلوبة. مستند مالي غير مقروء = غير قابل للاستخدام.

## القرار

**توليد PDF عبر Chromium (Playwright): قالب HTML → PDF.**

### لماذا محرك متصفح كامل؟

مكتبات PDF الخفيفة (`pdfkit`, `@react-pdf/renderer`, `jsPDF`) **لا تنفّذ تشكيل العربية**. تكتب الحروف كما وردت في السلسلة النصية، فتُنتج:

```
سلام   بدل   السلام
```

حروف منفصلة، غير مربوطة، وأحيانًا معكوسة الترتيب.

Chromium يملك محرك تخطيط نصّي كامل (HarfBuzz + ICU) — نفس المحرك الذي يعرض العربية والعبرية بشكل صحيح في كل صفحة ويب تراها. تشكيل صحيح، BiDi صحيح، أرباط صحيحة.

**وهو موجود أصلًا** كاعتمادية لاختبارات E2E — لا نضيف عبئًا جديدًا.

### قرارات مرافقة

| البند | القرار |
|---|---|
| الخطوط | **مُضمَّنة في الملف** (Noto Sans Arabic + Noto Sans Hebrew). PDF بلا خط مضمَّن يُعرض بخط بديل على جهاز الزبون — أو بمربّعات فارغة. |
| ثنائي اللغة | قالب واحد يعرض العنوانين معًا: «عرض سعر / הצעת מחיר». المتطلب 16 يفرضه. |
| الطباعة الحرارية | قالب CSS منفصل بعرض `80mm` + `@page`. لا PDF — طباعة مباشرة من المتصفح. |
| التخزين | كل مستند مولَّد يُحفظ مع `file_hash` — فالطلب المطبوع مرتين يعطي نفس الملف بالضبط، ونستطيع إثبات ما طُبع فعلًا. |
| الأداء | Chromium ثقيل (~200ms/مستند + ذاكرة). التوليد يجري في **طابور BullMQ** لا في مسار الطلب. |

## البدائل المرفوضة

| البديل | سبب الرفض |
|---|---|
| `pdfkit` / `jsPDF` | لا تشكيل عربي. عربية مقطّعة = مستند غير صالح. **حاسم.** |
| `@react-pdf/renderer` | نفس المشكلة (yoga + محرك نص بسيط). |
| LaTeX / XeTeX | تشكيل ممتاز، لكن سلسلة أدوات ضخمة وقوالب صعبة الصيانة على الفريق. |
| طباعة من المتصفح فقط | تعمل للشاشة، لكن لا تُنتج ملفًا قابلًا للأرشفة والإرسال بالبريد/واتساب. |
| خدمة PDF خارجية | بيانات مالية للزبائن تغادر خوادمنا. مرفوض. |

## العواقب

**إيجابية:** تشكيل عربي/عبري صحيح مضمون — نفس محرك المتصفح.

**سلبية:**
- استهلاك ذاكرة عالٍ ⇒ طابور + حد تزامن.
- حجم صورة النشر يكبر (~300MB لـChromium).
- **مخاطرة R6 مفتوحة:** يجب التحقق بلقطة بصرية للـPDF المولَّد في المرحلة 6، لا بالافتراض.
