# ADR-0004 — منع التسجيل المزدوج للدفعات (Idempotency)

**الحالة:** مُعتمد (التصميم) · **التنفيذ:** المرحلة 5 · **التاريخ:** 2026-07-14

---

## السياق

الكاشير يسجّل دفعة 500 ₪. الشبكة تتأخر. ينقر «حفظ» مرة أخرى. أو يضغط المتصفح إعادة الإرسال. أو تعيد مكتبة الشبكة المحاولة تلقائيًا بعد timeout — **بينما الطلب الأول نجح فعلًا على الخادم**.

النتيجة: دفعتان بـ500 ₪. رصيد الزبون ينقص 1000 ₪ بدل 500. خسارة مالية مباشرة، ونزاع مع الزبون.

هذا ليس سيناريو نادرًا — هو **الحالة الافتراضية** في أي شبكة حقيقية.

## القرار

**كل عملية مالية تتطلب مفتاح `Idempotency-Key` (UUID يولّده العميل).**

```
1. يصل POST /payments مع المفتاح K
2. INSERT INTO idempotency_keys (tenant_id, key, request_hash, status='IN_PROGRESS')
   → UNIQUE(tenant_id, key) يمنع التوازي ذرّيًا
3. موجود + COMPLETED   → أعِد الرد المخزَّن (200). بلا أي أثر جانبي.
4. موجود + hash مختلف  → 422. نفس المفتاح بحمولة مختلفة = خطأ عميل.
5. موجود + IN_PROGRESS → 409. الطلب الأول ما زال يعمل.
6. نجاح → خزّن الرد، status='COMPLETED'
```

**خط دفاع ثانٍ:** `UNIQUE(tenant_id, idempotency_key)` على `payments` و`ledger_entries` نفسيهما. حتى لو فشل منطق الطبقة الأولى، القاعدة ترفض.

### الواجهة: لا Optimistic UI للدفعات

قاعدة صريحة في المتطلبات، ونتبعها حرفيًا: الرصيد الجديد يُعرض **فقط** بعد تأكيد الخادم.

كذلك: `mutations.retry = false` في TanStack Query. إعادة إرسال `POST /payments` تلقائيًا وصامتة = دفعة مزدوجة محتملة. المستخدم يقرّر إعادة المحاولة، لا المكتبة.

وزر الحفظ يُعطَّل أثناء التنفيذ (`Button` يفرضها بنيويًا عبر `loading`).

## البدائل المرفوضة

| البديل | سبب الرفض |
|---|---|
| تعطيل الزر فقط | لا يحمي من إعادة إرسال المتصفح، ولا من إعادة محاولة الشبكة، ولا من نقرتين خلال ميلي‌ثانية. |
| كشف التكرار بالمحتوى (نفس المبلغ خلال دقيقة) | يرفض دفعتين شرعيتين متطابقتين. زبون يدفع 100 ₪ مرتين في نفس الدقيقة حالة حقيقية. |
| مفتاح من الخادم (GET ثم POST) | رحلة إضافية، ولا يحل إعادة إرسال POST نفسه. |

## العواقب

**إيجابية:** طلبان متزامنان بنفس المفتاح ⇒ دفعة واحدة. مضمون بقيد قاعدة بيانات، لا بمنطق تطبيق.

**سلبية:** جدول `idempotency_keys` ينمو ⇒ يحتاج تنظيفًا دوريًا (`expires_at` + مهمة BullMQ). العميل ملزم بتوليد المفتاح — موثّق في عقد الـAPI.
