# ADR-0003 — تمثيل الأموال

**الحالة:** مُعتمد · **التاريخ:** 2026-07-14 · **المرحلة:** 1

---

## السياق

JavaScript يمثّل كل الأرقام بـIEEE-754 double:

```js
0.1 + 0.2              // 0.30000000000000004
1.1 * 3                // 3.3000000000000003
0.07 * 100             // 7.000000000000001
(1.005).toFixed(2)     // "1.00"   ← وليس "1.01"
```

في نظام محاسبي هذا **قاتل**، لأن دفتر الحركات تراكمي: كل قيد يبني على `balance_after` السابق. خطأ 0.01 يدخل مرة واحدة، ثم يتضخّم عبر آلاف القيود، ولا يُكتشف إلا عند نزاع مع زبون.

## القرار

**المبلغ يعيش في ثلاث صور فقط، و`number` ليست إحداها.**

| الموضع | التمثيل |
|---|---|
| قاعدة البيانات | `NUMERIC(18,4)` |
| ذاكرة الخادم/الواجهة | `Decimal` (decimal.js) |
| **نقل عبر API** | **نص عشري** `"1250.00"` |
| العرض | نص منسّق `"1,250.00 ₪"` — لا يُعاد تحويله لرقم |

### لماذا نص عبر الـAPI؟

`JSON.parse('{"amount": 1250.10}')` يعطي `1250.1` كـdouble. القيم فوق `2^53` أو ذات كسور دقيقة تفقد الدقة **بلا إنذار**. النص يعبر الشبكة سليمًا ويُحوَّل إلى `Decimal` مباشرة على الطرفين.

### سياسة التقريب: HALF_UP

`1.005 → 1.01` دائمًا. رفضنا التقريب المصرفي (half-even) الذي يعطي `1.005 → 1.00`: صحيح إحصائيًا، لكنه يخالف عرف الفوترة وتوقّع التاجر، ويصعب شرحه في نزاع.

**التقريب يحدث في مكان واحد فقط:** `@oh/money`. لا يوجد `Math.round` ولا `.toFixed()` في أي مكان آخر.

### الفرض الآلي (لا اتفاق)

قواعد ESLint تمنع البناء عند:
- `.toFixed()` — إلا داخل `@oh/money` حيث هو `Decimal.prototype.toFixed` (دقيق).
- `parseFloat` / `Number.parseFloat`
- `Math.round`

النوع `MoneyString` يرفض `number` وقت الترجمة. ودالة `toMoney()` ترمي وقت التشغيل لو تسلّل رقم من JSON.

### التوزيع بلا فقدان فلس

`allocate()` — خوارزمية أكبر البواقي. توزيع 100 ₪ على 3 طلبات يعطي `33.34 + 33.33 + 33.33 = 100.00` بالضبط، لا `99.99`. مُختبَرة بـ500 حالة عشوائية.

## البدائل المرفوضة

| البديل | سبب الرفض |
|---|---|
| `number` + التقريب عند العرض | الخطأ يدخل الدفتر ويتراكم. التقريب عند العرض يُخفيه لا يُصلحه. |
| أعداد صحيحة (فلوس) | يعمل، لكن يكسر عند الكسور الوسيطة (ضريبة 17% على سطر) ويحتاج مضاعِفًا لكل عملية قسمة. |
| `BigInt` | لا كسور عشرية أصلًا. |
| `money` في Postgres | يعتمد على `lc_monetary` — نفس القيمة تُقرأ مختلفة حسب إعداد الخادم. |

## العواقب

**إيجابية:** يستحيل أن يمر مبلغ عبر `number`. مُثبَت بـ42 اختبارًا.

**سلبية:** كل تعامل مع مبلغ يمر بـ`@oh/money` — أطول قليلًا من `a + b`. هذا مقصود: الاحتكاك هنا **ميزة**.
